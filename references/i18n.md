# 多语言处理（i18n）

`hg_label` 文案词条化、批量加语言、混排文本与字体覆盖的完整流程。字段定义、JSON 结构与解析顺序以
**项目根 `HML-Spec.md`** 的 i18n 小节为准，本文只讲操作流程与易踩的坑。

## 机制速记

`hg_label` 加 `i18nKey` 关联 `i18n/strings.json` 词条，设计器多语言预览与 codegen 据此取**当前语言**
（`activeLocale`，回落 `defaultLocale`）的文案。`i18nKey` **只整串替换、无占位符插值**，缺 key/翻译时回落到 `text`。

- **仅 `hg_label` 生效**：写在 `hg_time_label` / `hg_timer_label` 或其他组件上会被解析但永不本地化
  （codegen 不翻、I18n Manager 不收录、预览不切换）。
- **改 `text` 对已绑定 `i18nKey` 的 label 无效** ⚠️：解析顺序是 catalog（当前/默认语言）→ `text`，
  只要 catalog 里有该 key 条目，改 `text=` 在预览和 codegen 里都不会有可见变化。要改文案得改
  `strings.json` 里对应词条的翻译（或换/清空 `i18nKey`），而不是改 `text`。

## 标签+动态数值混排：别整串词条化

像"行程: 126.5 km"这种整串，因 `i18nKey` 无插值，把数值烤进翻译等于把 mock 值当字面量发布。拆成两个 `hg_label`：

- 静态标签（"行程:"）用 `i18nKey`；
- 动态数值单独一个 label，配 `characterSets` 收集字形、**不绑 `i18nKey`**。

数字、单位（km/h、`%`）、日期、网络响应、用户输入等运行时文本一律走 `characterSets`，不建词条。

## 给已有工程批量加一种语言

agent 直接改 `strings.json` 是比人工逐条填表更快的主路径（这正是 vibe design 要发挥的地方）：

1. 往 `locales` 数组加新 locale；
2. 给 `strings` 里**每个 key** 补上新 locale 翻译，不留空洞；
3. 保留各词条默认语言 / `text` 作兜底；
4. **自检**：读回 `strings.json` 确认没有 key 缺新语言；若同时新增了词条，再 grep `ui/*.hml` 的 `i18nKey`
   与 json key 集合对账（查孤儿 key / 悬空引用）；
5. **字体覆盖** ⚠️：新语言可能带来当前字体没有的字形（日 / 韩 / 阿拉伯 / 西里尔文，或 `°C` 等符号）——
   引用这些 key 的 label 所用字体必须覆盖对应字符集，否则仿真渲染豆腐块，确认后再交付。

## I18n Manager（用户侧复核面）

设计器内置 **I18n Manager** 供用户可视化预览、按语言抽查缺失/孤儿/未绑定词条。它是 agent 批量直改
`strings.json` 的**补充复核面，不替代**直改——批量活仍由 agent 直接改 json 完成。
